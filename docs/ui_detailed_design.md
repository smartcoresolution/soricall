# SoriCall UI Detailed Design

작성일: 2026-07-06  
대상: SoriCall 모바일 Web/PWA 1차 구현, Android Compose 후속 구현

## 1. 문서 목적

본 문서는 안심소리 가족콜(SoriCall)의 서비스 시나리오를 바탕으로 실제 UI 화면을 설계하기 위한 세부 기준을 정의한다. 디자이너는 이 문서를 기준으로 Figma 화면을 제작하고, 개발자는 화면 상태, 입력값, API 연동, 예외 상태를 구현한다.

SoriCall은 어르신과 보호자가 함께 사용하는 가족 사칭 보이스피싱 대응 서비스다. UI는 "어르신이 위험 순간에 망설이지 않고 행동할 수 있는가"와 "보호자가 가족 확인 정보를 빠르게 등록하고 응답할 수 있는가"를 최우선 기준으로 삼는다.

## 2. 제품 UX 원칙

### 2.1 핵심 원칙

| 원칙 | 설명 |
|---|---|
| 큰 글자와 적은 선택지 | 어르신 화면은 한 화면의 주요 버튼을 1~2개로 제한한다. |
| 판단보다 행동 안내 | "위험 점수 92"보다 "전화를 끊고 저장된 가족 번호로 다시 전화하세요"를 우선 표시한다. |
| AI 결과는 참고 표현 | "사칭입니다"가 아니라 "가족 사칭 가능성이 높습니다"로 표현한다. |
| 개인정보 최소 노출 | 전화번호는 가능하면 끝자리만 표시한다. 음성 원본 보관 여부를 명확히 안내한다. |
| 보호자 응답 우선 | 위험 상황에서는 보호자 확인, 화상 확인, 재통화 안내가 즉시 보여야 한다. |
| 의료 오해 방지 | 치매 진단, 의료 판단, 법적 확정 표현을 사용하지 않는다. |

### 2.2 주요 사용자

| 사용자 | UI 목표 |
|---|---|
| 어르신 | 위험 전화 수신 시 안전 행동을 즉시 이해한다. |
| 보호자 | 부모님 연결, 가족 정보 등록, 위험 알림 응답을 빠르게 처리한다. |
| 가족 구성원 | 얼굴/음성 등록 동의와 샘플 등록 과정을 이해한다. |
| 관리자 | 위험번호와 이벤트를 확인하되 MVP UI에서는 우선순위가 낮다. |

## 3. 정보 구조

### 3.1 앱 전체 흐름

```text
Welcome
 ├─ Signup
 │   └─ Consent
 │       └─ Login
 └─ Login
     └─ Role Select
         └─ Family Connect
             └─ Home
                 ├─ Family
                 ├─ Guard
                 ├─ Verify
                 ├─ History
                 └─ Settings
```

### 3.2 위험 상황 흐름

```text
수신 전화
 └─ 번호 위험 평가
     ├─ 낮음: 일반 안내
     ├─ 주의: 노란 주의 안내
     ├─ 위험: 빨간 경고 + 보호자 알림
     └─ 매우 위험: 강한 경고 + 끊고 재통화 안내 + 보호자/화상 확인
```

### 3.3 가족 확인 흐름

```text
어르신이 "가족 확인" 요청
 └─ 보호자 알림
     ├─ 보호자: 내가 전화한 것 맞음
     ├─ 보호자: 내가 전화한 것 아님
     └─ 보호자: 모르겠음
         └─ 어르신 화면에 결과별 안내 표시
```

## 4. 공통 레이아웃

### 4.1 화면 기준

| 항목 | 기준 |
|---|---|
| 기본 대상 | 360px~480px 폭의 모바일 화면 |
| 최대 폭 | 480px 중앙 정렬 |
| 안전 영역 | iOS/Android safe area 고려 |
| 하단 내비게이션 | 로그인 및 가족 연결 후 노출 |
| 스크롤 | 본문 영역만 세로 스크롤 |

### 4.2 공통 헤더

구성:

- 왼쪽: 이전 버튼
- 중앙: 화면 제목
- 중앙 보조문구: 처리 상태, API 상태, 오류 메시지
- 오른쪽: 홈 버튼

상태 문구:

| 상태 | 표시 |
|---|---|
| 기본 | 완료 |
| 처리 중 | 처리 중 |
| API 확인 전 | API 확인 필요 |
| 오류 | 오류 메시지 직접 표시 |

### 4.3 하단 탭

| 탭 | 목적 | 아이콘 |
|---|---|---|
| 홈 | 현재 보호 상태 요약 | Home |
| 가족 | 가족 연락처, 얼굴, 음성 등록 | Users |
| 경고 | 전화번호 위험 평가, 보호자 알림 | Shield |
| 확인 | 보호자 응답, 화상 확인 | Video |
| 기록 | 위험 이벤트 이력 | History |

설정은 하단 탭에 넣지 않고 홈 또는 헤더/보조 버튼에서 접근한다. 어르신에게 반복 사용이 많은 기능을 탭에 우선 배치한다.

## 5. 시각 디자인 방향

### 5.1 톤

SoriCall은 금융 보안 앱처럼 차갑거나 의료 앱처럼 무거운 인상을 피한다. 가족 보호 서비스에 맞춰 따뜻하지만, 위험 상황에서는 명확하고 강한 대비를 사용한다.

| 영역 | 색상 방향 |
|---|---|
| 기본 배경 | 따뜻한 아이보리, 연한 녹색 |
| 주요 CTA | 청록색 계열 |
| 주의 | 노란색 |
| 위험 | 빨간색 |
| 정보 | 옅은 청록/민트 |
| 비활성 | 낮은 채도의 회색 |

### 5.2 타이포그래피

| 용도 | 권장 크기 |
|---|---:|
| 앱명/온보딩 제목 | 38~42px |
| 화면 제목 | 26~31px |
| 카드 제목 | 20~24px |
| 본문 | 16~18px |
| 보조 설명 | 13~15px |
| 어르신 위험 안내 | 24~34px |

한국어 기준 줄간격은 1.35~1.75를 사용한다. 위험 상황에서는 한 줄 문장을 짧게 유지한다.

### 5.3 버튼

| 버튼 | 용도 |
|---|---|
| Primary | 다음 단계, 로그인, 저장 |
| Danger | 위험 알림, 보호자 호출 |
| Warning | 주의 상태 확인 |
| Secondary | 보조 행동 |
| Ghost | 취소, 덜 중요한 이동 |

어르신 대상 화면의 주요 버튼 높이는 최소 56px 이상을 권장한다.

## 6. 화면별 세부 설계

### 6.1 Welcome

목적: 서비스 첫 인상과 진입 경로 제공.

주요 요소:

- 상단 회원가입 버튼
- 중앙 앱 아이콘과 `SoriCall` 로고
- 서비스 설명: "가족 사칭 전화를 감지하고 보호자 확인까지 이어주는 안심 통화 앱"
- 기본 CTA: `서비스 시작`
- 하단 API 연결 상태

주요 액션:

| 액션 | 이동 |
|---|---|
| 회원가입 | Signup |
| 서비스 시작 | Login |

빈/오류 상태:

- API 연결 실패 시 하단 notice에 오류 표시
- 오류가 있어도 회원가입/로그인 진입은 허용

### 6.2 Signup

목적: 보호자 또는 사용자 계정 생성 전 기본 정보 입력.

입력 필드:

| 필드 | 필수 | 검증 |
|---|---|---|
| 이름 | 필수 | 공백 불가 |
| 이메일 | 필수 | 이메일 형식 |
| 휴대폰 번호 | 선택 | 전화번호 키패드 |
| 비밀번호 | 필수 | 8자 이상 |
| 비밀번호 확인 | 필수 | 비밀번호와 일치 |

CTA:

- `가입하고 동의하기`
- `이미 계정이 있어요`

오류 문구:

- 이름, 이메일, 비밀번호를 입력해 주세요.
- 비밀번호는 8자 이상이어야 합니다.
- 비밀번호 확인이 일치하지 않습니다.

### 6.3 Consent

목적: 개인정보, 비의료 서비스, 음성/얼굴 분석 성격을 명확히 고지하고 동의를 받는다.

필수 동의:

| 항목 | 표시 문구 |
|---|---|
| 데이터 수집/분석 | 데이터 수집 및 분석에 동의합니다. |
| 개인정보 | 개인정보 처리 방침에 동의합니다. |
| 비의료 서비스 | 비의료적 서비스임을 이해하고 동의합니다. |
| 제3자 음성 가능성 | 제3자 음성 포함 가능성을 이해하고 동의합니다. |

선택 동의:

| 항목 | 표시 문구 |
|---|---|
| AI 연구 | AI 모델 개선 및 연구 활용에 동의합니다. |
| 알림/소식 | 서비스 소식과 보호 알림 안내 수신에 동의합니다. |

CTA:

- `동의하고 가입 완료`

상태:

- 필수 동의가 모두 체크되기 전에는 CTA 비활성
- 전체 동의 버튼은 필수/선택 모두 토글

### 6.4 Login

목적: 기존 계정으로 재진입.

입력 필드:

- 이메일
- 비밀번호

CTA:

- `로그인`

성공 이동:

- 가족 연결 데이터가 있으면 Home
- 없으면 Role Select

### 6.5 Role Select

목적: 사용자 역할에 따라 이후 연결 절차의 문맥을 정한다.

선택 카드:

| 카드 | 설명 |
|---|---|
| 부모님 전화 안심확인 | 보호자가 부모님 통화와 위험 알림을 함께 확인 |
| 내 안심 보호 시작 | 어르신 본인 전화와 가족 확인 요청을 보호자에게 연결 |

보조 요소:

- 이전 결과 보기
- 비의료 참고 서비스 안내 notice

선택 후 이동:

- Connect

### 6.6 Family Connect

목적: 가족 그룹, 어르신 프로필, 보호자 관계, 안심 단어의 초기 데이터를 구성한다.

입력 필드:

| 필드 | 역할 | 설명 |
|---|---|---|
| 가족 이름 | 공통 | 예: 김영희 가족 |
| 어르신 이름 | 공통 | 보호 대상 이름 |
| 어르신 휴대폰 | 공통 | 위험 평가 대상 번호 |
| 관계 | 보호자 | 딸, 아들, 배우자 등 |

백그라운드 생성:

- 가족 그룹 생성
- 어르신 프로필 생성
- 보호자 연결
- 기본 안심 단어 등록
- 데모 위험번호 등록
- 데모 가족 2명 생성
- 얼굴 프로필 생성
- 음성 프로필 생성 및 enroll

CTA:

- `연결하고 홈으로`

성공 이동:

- Home

### 6.7 Home

목적: 현재 보호 상태와 핵심 액션을 한 화면에서 제공한다.

상단 인사:

- 어르신 이름
- 연결된 계정 이름

상태 배너:

| 상태 | 문구 |
|---|---|
| 위험 없음 | 안심 상태 / 최근 위험 전화가 없습니다. |
| 위험 있음 | 위험 전화 확인 필요 / 최근 이벤트 요약 |

퀵 액션:

| 버튼 | 이동/동작 |
|---|---|
| 위험 전화 평가 | Guard에서 평가 실행 |
| 화상 가족 확인 | Verify 요청 생성 |
| 가족 등록 | Family |
| 안심 단어 | Settings |

요약 카드:

- 가족 연락처 수
- 얼굴 프로필 수
- 음성 프로필 수
- 최근 위험도

### 6.8 Family

목적: 가족 연락처, 얼굴 프로필, 음성 프로필을 관리한다.

섹션:

1. 연결된 가족 목록
2. 가족 추가 폼
3. 얼굴/음성 등록 상태

가족 목록 카드:

| 정보 | 표시 |
|---|---|
| 이름 | 전체 표시 |
| 관계 | 표시 |
| 전화번호 | 끝자리 4자리 중심 |
| 등록 상태 | 체크 아이콘 |

가족 추가 입력:

- 이름
- 관계
- 휴대폰 번호

CTA:

- `가족 추가`
- `목소리 등록`

개인정보 안내:

- 음성 등록은 명시적 동의와 사용자의 직접 조작이 필요하다는 문구를 하단에 둔다.

### 6.9 Guard

목적: 전화번호 위험 평가와 보호자 알림을 수행한다.

입력:

- 의심 전화번호

CTA:

- `위험도 평가`
- 평가 후 `보호자에게 알림`

결과 표시:

| 항목 | 설명 |
|---|---|
| 위험 단계 | LOW, MEDIUM, HIGH, CRITICAL 또는 한국어 표시 |
| 요약 | 위험 사유 요약 |
| 사유 코드 | 위험번호, 반복 전화, 심야 전화, 모르는 번호 등 |

위험도별 UI:

| 위험도 | 색/표현 | CTA |
|---|---|---|
| 낮음 | 청록/안심 | 기록 보기 |
| 주의 | 노랑 | 가족 확인하기 |
| 위험 | 빨강 | 보호자에게 알림 |
| 매우 위험 | 강한 빨강 | 전화를 끊고 가족 번호로 다시 전화 |

어르신 경고 문구 예:

```text
가족 사칭 가능성이 높습니다.
지금 통화 중이라면 전화를 끊고,
저장된 가족 번호로 다시 전화하세요.
```

### 6.10 Verify

목적: 보호자 응답과 화상 가족 확인 결과를 처리한다.

보호자 알림 응답:

| 버튼 | 값 |
|---|---|
| 내가 전화한 것 맞음 | REAL_CALL |
| 내가 전화한 것 아님 | NOT_ME |
| 모르겠음 | UNKNOWN |

응답별 어르신 안내:

| 응답 | 어르신 안내 |
|---|---|
| REAL_CALL | 가족이 확인한 전화입니다. 그래도 금전 요구가 있으면 다시 확인하세요. |
| NOT_ME | 가족이 전화하지 않았다고 답했습니다. 즉시 전화를 끊으세요. |
| UNKNOWN | 확인되지 않았습니다. 저장된 가족 번호로 다시 전화하세요. |

화상 확인:

| 상태 | 표시 |
|---|---|
| REQUESTED | 가족 확인 요청 중 |
| ACCEPTED | 확인 완료 |
| WAITING | 응답 대기 |

얼굴 비교 결과:

| 점수 | 표시 |
|---:|---|
| 80 이상 | 가족일 가능성이 높음 |
| 55~79 | 추가 확인 필요 |
| 55 미만 | 가족 확인 실패 가능성 |

주의:

- 얼굴 비교는 확정 인증이 아닌 참고 정보로 안내한다.
- 통화 중 금전 요구가 있으면 결과와 무관하게 재확인을 권장한다.

### 6.11 History

목적: 위험 이벤트와 보호자 알림 이력을 확인한다.

목록 항목:

| 정보 | 표시 |
|---|---|
| 발생 시각 | 상대 시간 또는 날짜 |
| 위험 단계 | 뱃지 |
| 요약 | 한 줄 |
| 전화번호 | 끝자리 또는 마스킹 |
| 처리 상태 | 알림 전송, 응답 대기, 확인 완료 |

상세 화면 후속 설계:

- 위험 사유
- 보호자 응답
- 화상 확인 결과
- 사용자가 취한 행동

빈 상태 문구:

```text
아직 위험 전화 기록이 없습니다.
```

### 6.12 Settings

목적: 안심 단어, 알림 옵션, API 상태, 로컬 데이터 초기화를 관리한다.

섹션:

| 섹션 | 항목 |
|---|---|
| 안심 단어 | 등록, 저장, 확인 |
| 보호 설정 | 모르는 번호 경고, 보호자 알림, 음성 경고 |
| 연결 정보 | API Base URL |
| 데이터 | 로컬 작업공간 초기화 |

안심 단어 결과:

| 상태 | 표시 |
|---|---|
| 일치 | 안심 단어가 확인되었습니다. |
| 불일치 | 안심 단어가 맞지 않습니다. |
| 미확인 | 확인 전 |

주의:

- 안심 단어 원문은 서버에 저장하지 않는다는 설명을 제공한다.
- 초기화는 로컬 데이터만 삭제하는지 서버 데이터도 삭제하는지 명확히 분리해야 한다.

## 7. 어르신 전용 위험 화면 설계

현재 웹앱의 Guard 화면을 기반으로, Android 또는 PWA 전체화면 알림에서는 더 단순한 어르신 전용 화면을 사용한다.

### 7.1 Suspicious Call Warning

목적: 통화 중 위험 신호가 감지되었을 때 즉시 행동을 유도한다.

화면 구성:

- 상단 큰 경고 아이콘
- 제목: `가족 사칭 의심 전화`
- 본문: `돈, 계좌, 앱 설치를 요구하면 전화를 끊으세요.`
- 주요 버튼: `전화를 끊고 가족에게 다시 전화`
- 보조 버튼: `보호자에게 확인 요청`

버튼 수:

- 위험/매우 위험 상태에서는 최대 2개

금지 표현:

- "100% 사기입니다"
- "AI가 확정했습니다"
- "치매 위험입니다"

### 7.2 Safe Word Check

목적: 상대방에게 가족 안심 단어를 물어보도록 안내한다.

화면 구성:

- 제목: `안심 단어를 물어보세요`
- 설명: `가족이라면 미리 정한 단어를 알고 있어요.`
- 입력 또는 선택: 어르신이 들은 단어 입력
- 결과: 일치/불일치

결과별 CTA:

| 결과 | CTA |
|---|---|
| 일치 | 그래도 돈 요구는 다시 확인 |
| 불일치 | 보호자에게 알림 |

## 8. 보호자 전용 알림 화면 설계

### 8.1 Guardian Push Landing

목적: 보호자가 알림을 눌렀을 때 바로 응답하게 한다.

제목:

```text
부모님에게 의심 전화가 왔습니다
```

표시 정보:

- 어르신 이름
- 전화번호 끝자리
- 위험 단계
- 발생 시각
- 위험 사유 요약

CTA:

| 버튼 | 우선순위 |
|---|---|
| 내가 전화한 것 아님 | 1 |
| 내가 전화한 것 맞음 | 2 |
| 모르겠음 | 3 |

위험한 경우 `내가 전화한 것 아님` 버튼을 가장 눈에 띄게 배치한다.

## 10. 상태 설계

### 10.1 공통 상태

| 상태 | UI |
|---|---|
| Loading | 버튼 비활성 + spinner + 헤더 `처리 중` |
| Success | 헤더 `완료`, 카드 데이터 갱신 |
| Error | 헤더 또는 폼 하단 오류 표시 |
| Empty | 친절한 빈 상태 문구 + 다음 액션 |
| Offline | API 확인 필요 + 재시도 버튼 |

### 10.2 입력 오류

오류는 필드 바로 아래 또는 카드 하단에 표시한다. 어르신 화면에서는 기술적 오류 대신 재시도 행동을 안내한다.

예:

```text
연결이 원활하지 않습니다.
잠시 후 다시 시도해 주세요.
```

## 11. 접근성

| 항목 | 기준 |
|---|---|
| 터치 영역 | 최소 44x44px, 주요 버튼 56px 이상 |
| 색 대비 | 본문 4.5:1 이상 |
| 글자 크기 | 본문 16px 이상 |
| 아이콘 | 아이콘만 쓰는 경우 aria-label 또는 텍스트 대체 |
| 색상 의존 | 위험 상태는 색상뿐 아니라 문구/아이콘으로도 표시 |
| 오류 안내 | 화면 낭독기가 읽을 수 있는 텍스트 제공 |

## 15. 구현 참고

```text
welcome, signup, consent, login, role, connect,
```

- 안심 단어는 hash 저장
- AI 판단은 확정이 아닌 위험 가능성 안내
