# 안심소리 가족콜(SoriCall) UI 디자인 설계서 - 수정 보완본

작성일: 2026-07-06  
대상: SoriCall 모바일 Web/PWA 1차 구현, Android Compose 후속 구현  
기준 문서: `ui_detailed_design.md`  
문서 목적: 현재 구현된 화면 구조를 유지하면서, 가족 얼굴/음성/화상 확인 중심의 어르신 친화 UI를 보완하여 Figma 설계, 프론트엔드 구현, API 연동, 테스트 검수에 바로 사용할 수 있는 기준을 제시한다.

## 1. 수정 보완 요약

현재 첨부 설계서는 `welcome, signup, consent, login, role, connect, home, family, guard, verify, history, settings` 화면 구조와 API 연동이 이미 잘 정리되어 있다. 따라서 전체 구조는 유지하고, 다음 부분을 보완한다.

| 구분 | 현재 설계의 장점 | 보완 필요 사항 | 보완 방향 |
|---|---|---|---|
| 정보 구조 | 현재 코드 화면명과 흐름이 명확함 | 어르신 전용 홈과 보호자 홈의 시각적 차이가 더 필요함 | 동일 Home 화면 안에 `Senior Home Variant`와 `Guardian Home Variant`를 분리 정의 |
| 가족 등록 | 가족 연락처, 얼굴, 음성 프로필 구조가 있음 | 가족 얼굴 카드, 음성 등록 품질, 동의 절차가 더 구체화될 필요 | 가족 카드 컴포넌트와 얼굴/음성 등록 단계 추가 |
| 위험 전화 | Guard, Verify 흐름이 있음 | 실제 수신 전화 오버레이와 통화 중 경고 화면이 2차로만 제시됨 | MVP 데모에서는 시뮬레이션, Android/PWA에서는 전체화면 위험 오버레이로 확장 |
| 화상 확인 | Verify에 얼굴 비교 결과가 있음 | “등록된 얼굴과 현재 통화 얼굴 비교” 흐름이 화면 단위로 부족 | Video Verify 화면과 보호자 Push Landing을 상세화 |
| 어르신 접근성 | 큰 글자, 적은 버튼 원칙이 있음 | 실제 버튼/문구/색상 토큰이 더 필요 | 디자인 토큰, 상태별 문구, 컴포넌트 사양 추가 |
| 개인정보 | 개인정보 최소 노출, AI 참고 표현이 있음 | 생체정보 동의와 원본 미보관 정책의 화면 반영 필요 | 음성/얼굴 등록 전 별도 동의 카드 추가 |
| 개발 인수 | API 매핑이 있음 | 화면 상태, 이벤트명, 인수 기준이 부족 | 화면별 상태, 이벤트 로그, QA 체크리스트 추가 |

## 2. 제품 정의

### 2.1 서비스명

| 항목 | 명칭 |
|---|---|
| 플랫폼/서비스군 | 안심소리 AI |
| 서비스명 | 안심소리 가족콜 |
| 앱명 | SoriCall |
| 핵심 가치 | 가족 사칭 보이스피싱 예방, 가족 확인, 보호자 알림, 안전 재통화 |

### 2.2 한 문장 정의

SoriCall은 어르신에게 걸려온 전화가 실제 가족인지 전화번호, 가족 안심 단어, 보호자 응답, 음성 등록 정보, 얼굴/화상 확인 정보를 종합하여 안내하는 가족 보호형 안심 통화 서비스이다.

### 2.3 MVP 범위

| 범위 | 포함 여부 | 설명 |
|---|---:|---|
| 회원가입/로그인 | 포함 | 현재 화면 구조 유지 |
| 가족 그룹 연결 | 포함 | 보호자-어르신 관계 설정 |
| 가족 연락처 등록 | 포함 | 이름, 관계, 전화번호, 등록 상태 |
| 얼굴 프로필 등록 | 포함 | 사진/촬영 기반 등록 UI |
| 음성 프로필 등록 | 포함 | 명시적 동의 후 샘플 녹음/업로드 |
| 위험 전화번호 평가 | 포함 | Guard 화면에서 평가 |
| 보호자 알림 응답 | 포함 | Verify 화면에서 REAL_CALL / NOT_ME / UNKNOWN |
| 화상 가족 확인 | 포함 | MVP에서는 요청/수락/결과 표시, 후속 구현에서 실제 영상 연결 |
| 실제 통화 가로채기/몰래 녹음 | 제외 | 정책·기술·동의 이슈로 MVP 제외 |
| 의료 진단 | 제외 | 치매 진단, 건강 판단 표현 금지 |

## 3. 핵심 UX 원칙

### 3.1 어르신 화면 원칙

| 원칙 | 설계 기준 |
|---|---|
| 한 화면 한 행동 | 위험 상황에서는 `끊기`, `가족에게 다시 전화`, `보호자 확인` 중 최대 2개만 표시 |
| 큰 얼굴 중심 | 가족 식별은 이름보다 얼굴 카드가 먼저 보이도록 설계 |
| 쉬운 문장 | “AI confidence 0.83” 대신 “가족일 가능성이 높습니다” 사용 |
| 행동 우선 | 분석 결과보다 “지금 무엇을 해야 하는가”를 먼저 표시 |
| 반복 안내 | 위험 문구는 화면 텍스트, 색상, 아이콘, 음성 안내로 동시에 제공 |
| 확정 표현 금지 | “사기 확정”, “범인”, “100% 가족 아님” 금지 |

### 3.2 보호자 화면 원칙

| 원칙 | 설계 기준 |
|---|---|
| 즉시 응답 | 위험 알림을 누르면 바로 `내가 전화한 것 아님` 버튼이 보이도록 함 |
| 등록 상태 가시화 | 가족별 얼굴/음성/전화번호 등록 완료 여부를 한눈에 표시 |
| 로그 확인 | 위험 이벤트의 발생 시각, 사유, 응답 상태를 확인 가능하게 함 |
| 수치와 문장 병행 | 보호자에게는 일치율 수치와 쉬운 해석을 함께 표시 가능 |

### 3.3 가족 구성원 화면 원칙

| 원칙 | 설계 기준 |
|---|---|
| 동의 기반 등록 | 얼굴/음성 등록 전 목적, 보관 여부, 삭제 방법을 명확히 안내 |
| 짧은 등록 절차 | 음성 등록은 3~5개 문장, 30초 이내 완료를 목표로 함 |
| 재등록 가능 | 품질 낮음, 소음 많음, 얼굴 흐림 상태에서는 재등록 CTA 제공 |

## 4. 현재 코드 기준 정보 구조

현재 구현 화면명은 다음 구조를 유지한다.

```text
welcome, signup, consent, login, role, connect,
home, family, guard, verify, history, settings
```

### 4.1 전체 흐름

```text
Welcome
 ├─ Signup
 │   └─ Consent
 │       └─ Login
 └─ Login
     └─ Role Select
         └─ Family Connect
             └─ Home
                 ├─ Family
                 ├─ Guard
                 ├─ Verify
                 ├─ History
                 └─ Settings
```

### 4.2 보완 흐름: 실제 수신 전화 대응

```text
수신 전화 또는 의심 전화 입력
 └─ Guard: 번호 위험 평가
     ├─ LOW: 일반 안내 + 기록 저장
     ├─ MEDIUM: 안심 단어 확인 권장
     ├─ HIGH: 보호자 알림 + 가족 재통화 권장
     └─ CRITICAL: 강한 경고 + 통화 종료 권장 + 보호자/화상 확인 요청
```

### 4.3 보완 흐름: 가족 얼굴/화상 확인

```text
어르신: 가족 확인 요청
 └─ 보호자/가족 앱 Push 또는 알림
     ├─ 가족: 내가 전화한 것 맞음
     ├─ 가족: 내가 전화한 것 아님
     ├─ 가족: 모르겠음
     └─ 필요 시 화상 확인 요청
         ├─ 등록된 가족 얼굴 프로필 표시
         ├─ 현재 영상 얼굴 캡처/프레임 비교
         └─ 어르신 화면에 쉬운 문장으로 결과 표시
```

> 구현 주의: “자동 화상 연결”은 사전 동의, 카메라 권한, 푸시 알림, 앱 간 딥링크, OS 정책에 영향을 받는다. MVP 문구는 “화상 확인 요청” 또는 “가족 화상 확인 연결”로 표현하고, 실제 서비스에서는 사용자 동의 기반의 원터치 연결을 우선 적용한다.

## 5. 사용자별 화면 분리 기준

### 5.1 어르신 모드

| 화면 | 목적 | 핵심 UI |
|---|---|---|
| Home | 가족에게 쉽게 전화, 보호 상태 확인 | 가족 얼굴 카드, 큰 전화 버튼, 긴급 도움 |
| Guard | 위험 전화 안내 | 큰 경고 문구, 끊기/재통화 버튼 |
| Verify | 가족 확인 결과 확인 | 보호자 응답, 얼굴/화상 확인 결과 |
| History | 최근 위험 전화 확인 | 단순 목록, 안전/주의/위험 배지 |
| Settings | 글자 크기, 음성 안내, 보호자 정보 | 최소 메뉴 |

### 5.2 보호자 모드

| 화면 | 목적 | 핵심 UI |
|---|---|---|
| Home | 부모님 보호 상태 요약 | 위험 이벤트, 가족 등록 현황, 최근 통화 |
| Family | 가족 연락처/얼굴/음성 관리 | 가족 카드, 등록 상태, 추가/수정 |
| Guard | 의심 번호 평가 및 알림 | 번호 입력, 위험 사유, 알림 전송 |
| Verify | 보호자 응답 | REAL_CALL / NOT_ME / UNKNOWN, 화상 확인 |
| History | 이벤트 이력 | 상세 로그, 필터, 처리 상태 |
| Settings | 안심 단어, 알림, API 상태 | 보호 정책 설정 |

## 7. 핵심 컴포넌트 설계

### 7.1 Family Face Card

목적: 어르신이 이름보다 얼굴과 관계로 가족을 인식하도록 돕는 핵심 카드.

| 요소 | 설명 |
|---|---|
| 얼굴 이미지 | 원형 또는 둥근 사각형, 88~120px 이상 |
| 이름 | 예: 김민수 |
| 관계 | 예: 아들, 딸, 배우자 |
| 전화번호 | 보호자 모드에서는 끝 4자리, 어르신 모드에서는 숨김 가능 |
| 등록 배지 | 얼굴 등록 완료, 음성 등록 완료, 안심 단어 등록 |
| 주요 버튼 | 전화하기, 영상 확인, 다시 확인 |

상태별 배지:

| 상태 | 배지 문구 | 색상 |
|---|---|---|
| 모든 등록 완료 | 안심 등록 완료 | safe |
| 얼굴만 등록 | 얼굴 등록 완료 | primary |
| 음성만 등록 | 목소리 등록 완료 | primary |
| 미등록 | 등록 필요 | caution |
| 품질 낮음 | 다시 등록 권장 | warning |

### 7.2 Safety Status Badge

| 위험도 | 내부 값 | 어르신 문구 | 보호자 문구 |
|---|---|---|---|
| 낮음 | LOW | 안전해 보입니다 | 낮음 |
| 주의 | MEDIUM | 한 번 더 확인하세요 | 주의 |
| 위험 | HIGH | 가족 사칭 가능성이 있습니다 | 위험 |
| 매우 위험 | CRITICAL | 지금 전화를 끊으세요 | 매우 위험 |
| 확인 불가 | UNKNOWN | 확인이 어렵습니다 | 확인 불가 |

### 7.3 Action Button

| 유형 | 문구 예 | 사용 위치 |
|---|---|---|
| Primary | 가족에게 전화하기 | Home, Family |
| Danger | 지금 전화 끊기 | Guard, Suspicious Call Warning |
| Warning | 가족 확인하기 | Guard |
| Secondary | 보호자에게 확인 요청 | Verify |
| Ghost | 나중에 하기 | 가입/설정 |

### 7.4 Warning Banner

위험 상태에서는 카드 안쪽의 작은 메시지가 아니라 화면 상단 또는 중앙에 큰 배너로 표시한다.

```text
가족 사칭 가능성이 높습니다.
돈, 계좌번호, 비밀번호를 말하지 마세요.
저장된 가족 번호로 다시 전화하세요.
```

## 8. 화면별 세부 설계 보완

## 8.1 Welcome

### 목적

서비스 첫 인상과 진입 경로 제공.

### 주요 구성

- 상단: 회원가입
- 중앙: SoriCall 로고, 앱 아이콘
- 설명: “가족 사칭 전화를 감지하고 보호자 확인까지 이어주는 안심 통화 앱”
- 기본 CTA: 서비스 시작
- 하단: API 연결 상태

### 보완 사항

| 항목 | 보완 내용 |
|---|---|
| 신뢰 문구 | “AI 분석 결과는 참고 정보이며, 위험 시 저장된 가족 번호로 재확인하세요.” |
| 어르신 안내 | 첫 화면에서 너무 많은 기능을 설명하지 않고 2문장 이내로 제한 |
| API 오류 | 오류가 있어도 로그인/회원가입 진입은 허용 |

## 8.2 Signup

### 목적

보호자 또는 사용자 계정 생성.

### 입력 필드

| 필드 | 필수 | 검증 |
|---|---:|---|
| 이름 | 필수 | 공백 불가 |
| 이메일 | 필수 | 이메일 형식 |
| 휴대폰 번호 | 선택 | 숫자/하이픈 허용 |
| 비밀번호 | 필수 | 8자 이상 |
| 비밀번호 확인 | 필수 | 비밀번호와 일치 |

### 보완 사항

- 휴대폰 번호는 선택으로 두되, 보호자 알림 기능 사용 전에는 필수 등록을 요구한다.
- 가입 완료 후 바로 생체정보 등록을 강제하지 않고 Consent를 먼저 보여준다.

## 8.3 Consent

### 목적

개인정보, 음성/얼굴 정보, 비의료 서비스, AI 참고 분석의 성격을 명확히 고지한다.

### 필수 동의

| 항목 | UI 문구 |
|---|---|
| 개인정보 수집/이용 | 서비스 제공을 위한 개인정보 수집 및 이용에 동의합니다. |
| 음성 정보 처리 | 가족 확인을 위한 음성 특징 분석에 동의합니다. |
| 얼굴 정보 처리 | 가족 확인을 위한 얼굴 이미지/특징 분석에 동의합니다. |
| 비의료 서비스 | 이 서비스는 의료 진단을 제공하지 않음을 이해했습니다. |
| AI 참고 분석 | AI 결과는 참고 정보이며 최종 판단이 아님을 이해했습니다. |

### 선택 동의

| 항목 | UI 문구 |
|---|---|
| 모델 개선 | 익명화된 품질 정보의 AI 모델 개선 활용에 동의합니다. |
| 알림/소식 | 서비스 안내와 보호 알림 수신에 동의합니다. |

### 보완 사항

- 음성/얼굴 정보는 일반 개인정보보다 민감하게 느껴질 수 있으므로 별도 카드로 분리한다.
- “언제든 삭제 가능” 문구를 동의 화면과 Settings 화면에 반복 표시한다.

## 8.5 Role Select

## 8.6 Family Connect

### 목적

가족 그룹, 어르신 프로필, 보호자 관계, 기본 안심 단어를 구성한다.

### 입력 필드

| 필드 | 역할 | 설명 |

| 가족 이름 | 공통 | 예: 김영희 가족 |
| 어르신 이름 | 공통 | 보호 대상 이름 |
| 어르신 휴대폰 | 공통 | 위험 평가 대상 번호 |
| 관계 | 보호자 | 딸, 아들, 배우자 등 |

### 백그라운드 생성

- 가족 그룹 생성
- 어르신 프로필 생성
- 보호자 연결
- 기본 안심 단어 등록
- 데모 위험번호 등록
- 데모 가족 2명 생성
- 얼굴 프로필 생성
- 음성 프로필 생성 및 enroll

### 보완 사항

데모 가족 2명은 개발/시연용으로만 사용하고, 실제 서비스에서는 사용자가 명시적으로 가족을 추가하도록 한다.

## 8.7 Home

### 목적

현재 보호 상태와 핵심 액션을 한 화면에서 제공한다.

### Senior Home Variant

어르신 홈은 가족 얼굴 중심으로 구성한다.

```text
┌────────────────────────┐
│ 안심소리 가족콜          │
│ 오늘도 안전하게 통화하세요 │
├────────────────────────┤
│ [아들 얼굴] 김민수 / 아들 │
│ [전화하기] [영상확인]     │
├────────────────────────┤
│ [딸 얼굴] 김지연 / 딸    │
│ [전화하기] [영상확인]     │
├────────────────────────┤
│ [보호자에게 도움 요청]    │
└────────────────────────┘
```

#### Senior Home 요소

| 영역 | UI |
|---|---|
| 상단 | 보호 상태 문구: “안심 보호 중입니다” |
| 중앙 | 가족 얼굴 카드 2~4개 우선 노출 |
| 주요 CTA | 가족에게 전화하기 |
| 보조 CTA | 보호자에게 도움 요청 |
| 하단 | 최근 통화, 설정 |

### Guardian Home Variant

보호자 홈은 관리/요약 중심으로 구성한다.

| 영역 | UI |
|---|---|
| 상단 | 어르신 이름, 보호 상태 |
| 요약 카드 | 가족 등록 수, 얼굴 등록 수, 음성 등록 수 |
| 위험 카드 | 최근 위험 전화, 미응답 알림 |
| 퀵 액션 | 가족 등록, 위험 평가, 화상 확인 요청 |

### 보완 사항

- Home 화면은 하나의 코드 컴포넌트로 유지하되 `role` 값에 따라 카드 우선순위를 다르게 렌더링한다.
- 어르신 화면에는 수치를 최소화하고, 보호자 화면에는 등록률과 위험도 요약을 표시한다.

## 8.8 Family

### 목적

가족 연락처, 얼굴 프로필, 음성 프로필을 관리한다.

### 가족 목록 카드

| 정보 | 어르신 모드 | 보호자 모드 |
|---|---|---|
| 얼굴 | 크게 표시 | 중간 크기 표시 |
| 이름/관계 | 크게 표시 | 일반 표시 |
| 전화번호 | 숨김 또는 끝 4자리 | 끝 4자리 |
| 등록 상태 | “안심 등록 완료” | 얼굴/음성/번호별 체크 |
| 액션 | 전화하기 | 수정, 재등록, 삭제 |

### 가족 추가 입력

- 이름
- 관계
- 휴대폰 번호
- 얼굴 등록
- 목소리 등록
- 영상 확인 사용 여부

### 보완 사항: 얼굴 등록

| 단계 | UI |
|---|---|
| 1 | 얼굴 등록 목적 안내 |
| 2 | 사진 선택 또는 카메라 촬영 |
| 3 | 얼굴 감지 여부 확인 |
| 4 | 흐림/역광/가림 상태 점검 |
| 5 | 등록 완료 또는 재촬영 |

### 보완 사항: 음성 등록

| 단계 | UI |
|---|---|
| 1 | 음성 등록 동의 |
| 2 | 조용한 장소 안내 |
| 3 | 3~5개 문장 녹음 |
| 4 | 소음/길이/볼륨 품질 확인 |
| 5 | 등록 완료 또는 다시 녹음 |

권장 녹음 문장 예:

```text
1. 안심소리 가족콜 목소리 등록을 시작합니다.
2. 가족 확인을 위해 제 목소리를 등록합니다.
3. 전화를 받기 전에 꼭 가족인지 확인해 주세요.
4. 돈을 보내기 전에는 반드시 다시 전화해 주세요.
5. 오늘도 안전하게 통화하세요.
```

## 8.9 Guard

### 목적

전화번호 위험 평가와 보호자 알림을 수행한다.

### 입력

- 의심 전화번호
- 통화 중 여부
- 상대방이 주장한 가족 관계
- 금전/계좌/앱 설치 요구 여부
- 모르는 번호 여부

### 결과 표시

| 항목 | 설명 |
|---|---|
| 위험 단계 | LOW, MEDIUM, HIGH, CRITICAL |
| 어르신용 문구 | 쉬운 행동 안내 |
| 보호자용 요약 | 위험 사유와 로그 |
| CTA | 가족 확인, 보호자 알림, 저장된 번호로 재통화 |

### 위험도별 UI

| 위험도 | 어르신 문구 | 주요 CTA |
|---|---|---|
| LOW | 안전해 보입니다. 그래도 돈 요구는 다시 확인하세요. | 기록 보기 |
| MEDIUM | 한 번 더 확인이 필요합니다. | 안심 단어 확인 |
| HIGH | 가족 사칭 가능성이 있습니다. | 보호자에게 확인 요청 |
| CRITICAL | 지금 전화를 끊고 저장된 가족 번호로 다시 전화하세요. | 전화 끊기 / 가족에게 다시 전화 |

### 보완 사항

Guard 화면은 개발/시연 단계에서 “전화번호 입력 기반 위험 평가”로 동작하고, Android/PWA 후속 구현에서는 실제 수신 전화 알림 오버레이와 연결한다.

## 8.10 Verify

### 목적

보호자 응답과 화상 가족 확인 결과를 처리한다.

### 보호자 응답

| 버튼 | 값 | 어르신 화면 문구 |
|---|---|---|
| 내가 전화한 것 맞음 | REAL_CALL | 가족이 확인한 전화입니다. 그래도 돈 요구가 있으면 다시 확인하세요. |
| 내가 전화한 것 아님 | NOT_ME | 가족이 전화하지 않았다고 답했습니다. 즉시 전화를 끊으세요. |
| 모르겠음 | UNKNOWN | 확인되지 않았습니다. 저장된 가족 번호로 다시 전화하세요. |

### 화상 확인 상태

| 상태 | 표시 |
|---|---|
| REQUESTED | 가족 확인 요청 중 |
| WAITING | 응답 대기 |
| ACCEPTED | 화상 확인 연결됨 |
| COMPARED | 얼굴 비교 완료 |
| FAILED | 확인 실패 |

### 얼굴 비교 결과 표현

| 점수 | 보호자 화면 | 어르신 화면 |
|---:|---|---|
| 80 이상 | 얼굴 일치율 높음 | 등록된 가족일 가능성이 높습니다 |
| 55~79 | 추가 확인 필요 | 한 번 더 확인하세요 |
| 55 미만 | 가족 확인 실패 가능성 | 가족이 아닐 수 있습니다 |
| 확인 불가 | 카메라/조명/네트워크 문제 | 얼굴 확인이 어렵습니다 |

### 보완 사항

- 어르신 화면에는 “얼굴 일치율 83%” 대신 “등록된 가족일 가능성이 높습니다”로 표시한다.
- 보호자 화면에는 수치와 사유를 함께 표시한다.
- 금전 요구가 있는 경우 얼굴 결과와 무관하게 재통화를 권장한다.

## 8.11 History

### 상세 화면 보완

- 위험 사유
- 보호자 응답

- 얼굴/음성 확인 결과

- 재통화 여부
- 알림 전송 상태

## 8.12 Settings

### 섹션

|---|---|
| 안심 단어 | 등록, 변경, 확인 |

### 보완 사항

- “로컬 작업공간 초기화”와 “서버 데이터 삭제”를 명확히 분리한다.

## 9. 실제 수신 전화 오버레이 설계

현재 코드가 모바일 Web/PWA 중심이라면 실제 통화 수신을 직접 제어하기 어렵다. 따라서 MVP와 후속 구현을 분리한다.

### 9.1 MVP/PWA 시뮬레이션 화면

| 항목 | 설계 |
|---|---|
| 진입 | Guard 화면에서 의심 번호 입력 |
| 위험 판단 | API 평가 결과 표시 |
| 보호자 알림 | emergency notify API 호출 |
| 화상 확인 | Verify 요청 생성 |
| 기록 | History 저장 |

### 9.2 Android 후속 전체화면 알림

```text
┌────────────────────────┐
│ 가족 사칭 의심 전화      │
├────────────────────────┤
│ 돈, 계좌, 앱 설치를 요구하면 │
│ 전화를 끊으세요.          │
├────────────────────────┤
│ [지금 전화 끊기]          │
│ [보호자에게 확인 요청]     │
└────────────────────────┘
```

### 9.3 오버레이 상태

| 상태 | 문구 | CTA |
|---|---|---|
| 확인 중 | 가족인지 확인하고 있습니다 | 잠시 기다리기 |
| 주의 | 한 번 더 확인이 필요합니다 | 안심 단어 묻기 |
| 위험 | 가족 사칭 가능성이 있습니다 | 보호자 확인 요청 |
| 매우 위험 | 지금 전화를 끊으세요 | 전화 끊기 / 가족에게 다시 전화 |

## 10. 화상 확인 UI 상세

### 10.1 어르신 화면

| 영역 | UI |
|---|---|
| 상단 | “가족 얼굴을 확인하고 있습니다” |
| 중앙 | 등록된 가족 얼굴 + 현재 영상 화면 |
| 결과 | 쉬운 문장으로 표시 |
| CTA | 통화 계속 / 보호자에게 확인 / 끊기 |

- [ ] 음성/얼굴 등록 전 동의 문구가 표시된다.
- [ ] AI 결과가 확정 표현으로 표시되지 않는다.

- [ ] 보호자 Push Landing은 `내가 전화한 것 아님` 버튼이 가장 잘 보인다.

## 20. 결론

현재 첨부 설계서는 실제 코드 화면명, API 매핑, 위험 흐름, 접근성 원칙이 잘 정리되어 있으므로 큰 구조를 변경할 필요는 없다. 다만 SoriCall의 핵심 차별점은 “가족 얼굴과 목소리를 기반으로 어르신이 위험 순간에 즉시 행동하도록 돕는 것”이므로, 보완 설계에서는 다음 5가지를 강화하였다.

1. 어르신 홈을 가족 얼굴 카드 중심으로 재정의
2. 보호자 홈을 등록 상태와 위험 알림 중심으로 분리
3. 가족 얼굴/음성 등록 품질 기준과 동의 UI 추가
4. 실제 수신 전화 오버레이와 화상 확인 흐름 상세화
5. 개발 인수 기준과 QA 체크리스트 추가

따라서 최종 UI 설계는 현재 Web/PWA 코드 구조를 유지하면서도, 향후 Android Compose와 실제 통화 알림 기능까지 확장 가능한 구조로 설계하는 것이 적절하다.
